# 美化规则编辑器 — 设计规格

<!-- Last updated: 2026-07-24 -->

## 概述

在 LCTA 主应用中新增一个独立的 pywebview 窗口——美化规则编辑器，从"文本美化"页面触发打开。提供文件浏览、规则编辑（简单/高级双模式）、智能规则生成三大核心功能。

### 动机

- 当前用户自定义美化规则以 JSON 字符串存储在 `ConfigManager.user_fancy` 中，编辑体验差（纯文本 textarea），不利于管理
- 用户需要可视化地浏览游戏 JSON 文件内容，在不懂正则和 trigger 语法的情况下创建规则
- 规则需要支持基于 ID 的稳定定位，使规则在游戏更新后依然有效

## 二、规则文件格式

### 2.1 规则集文件（`fancy/*.json`）

每个文件一个规则集，纯 JSON 格式：

```json
{
  "name": "我的技能美化",
  "desc": "替换技能描述中的符号并着色",
  "rules": [
    {
      "aimFile": "Skill.*\\.json$",
      "conditions": [
        {
          "trigger": { "aim": "dataList\\.\\d+\\.id", "re": "^10001$" },
          "aim": "[back].name"
        }
      ],
      "action": [
        { "from": "大于", "to": ">" },
        { "from": "自身", "to": "<u><color=#7C5738>自身</color></u>" }
      ]
    }
  ]
}
```

### 2.2 `conditions` 新格式（与旧版兼容）

#### 旧版格式（自动转换为 conditions）

```json
// 有 trigger 的旧格式
{ "trigger": {...}, "aim": "...", "action": [...] }

// 等价为
{ "conditions": [{ "trigger": {...}, "aim": "..." }], "action": [...] }

// 无 trigger 的旧格式
{ "aim": "dataList\\.\\d+\\.desc", "action": [...] }

// 等价为
{ "conditions": [{ "aim": "dataList\\.\\d+\\.desc" }], "action": [...] }
```

#### 新 conditions 格式

```json
{
  "aimFile": "Skill.*\\.json$",
  "conditions": [
    {
      "trigger": { "aim": "dataList\\.\\d+\\.id", "re": "^10001$" },
      "aim": "[back].name",
      "fallback": { "index": 0 }
    },
    {
      "trigger": { "aim": "dataList\\.\\d+\\.desc", "re": "指定" },
      "aim": "[back].name"
    }
  ],
  "action": [...]
}
```

**语义**：所有 conditions 都满足的同一个 `dataList[N]` 条目才命中，对每个 condition 的 `aim` 目标字段应用 `action`。

**`fallback` 字段**（可选）：当 trigger 按 id 匹配失败时，回退到指定索引。仅在自动生成规则时由系统填充，用户可手动编辑。

### 2.3 exec_json 改造

```
对每个 rule：
  若存在 trigger/aim（旧格式）→ 内部转为 conditions: [{trigger, aim}]
  若存在 conditions（新格式）→ 直接使用

  取第一个 condition 的 trigger.aim 匹配所有路径
  → 对每个匹配路径，检查所有 conditions：
      每个 condition 的 trigger.aim 匹配当前条目 ✓
      trigger.re 匹配对应值 ✓
      condition.aim 指向的目标字段存在 ✓
  → 全部通过 → 对每个 condition.aim 指向的字段应用 action
```

### 2.4 迁移方案

首次启动新版时，检测 `ConfigManager.user_fancy` 是否还有旧数据：
- 有 → 自动导出为 `fancy/` 下独立 JSON 文件，清空 config 中的旧字段
- 无 → 跳过

## 五、智能生成规则集

### 5.1 触发方式

用户在文件浏览器中做了一系列修改后，点击"🧠 智能生成规则集"按钮。

### 5.2 算法流程

```
输入: 用户修改列表 [(file, field, item_id, old, new), ...]

Phase 1: 值变化分析
  for each modification:
    1. LCS分析 → 分解 prefix_added, core_old, core_new, suffix_added
    2. 分类: PURE_REPLACE | PURE_WRAP | REPLACE_WRAP | PREFIX_ONLY | SUFFIX_ONLY
    3. 解析 HTML 标签 → 抽象样式模板 {tag, color, ...}

Phase 2: 模式聚类
  1. REPLACE 类 → 按 (core_old, core_new) 聚类
  2. WRAP 类 → 按 (prefix_template, suffix_template) 聚类
  3. 每组分析文件/条目/字段分布

Phase 3: 范围推断 (L1-L4)
  L1 文件范围: 单文件→精确 / 同分类→分类 / 跨分类→多分类正则
  L2 条目定位: 有id→id定位 / 无id→全文匹配+索引fallback
  L3 字段约束: 同字段→限定 / 跨字段→所有文本字段
  L4 额外条件: 默认完整匹配

Phase 4: 五维评分（详见 5.3）

Phase 5: 后处理
  - 跨组合并检测（L1+L2+L3+L4 完全相同 → 建议合并）
  - 规律总结（自然语言描述用户意图）
  - 补充建议（检测到不完整模式时推荐添加）

Phase 6: 排序展示
  按 priority 降序，标签: 🟢推荐(≥70) | 🟡可选(40-69) | 🔴需审查(<40)
```

### 5.3 五维评分体系（待细化）

| 维度 | 权重 | 评估方向 |
|------|------|---------|
| S1 覆盖广度 | 0.25 | 影响文件数、条目数、出现次数 |
| S2 模式纯度 | 0.20 | 修改模式一致性，异常值惩罚 |
| S3 可泛化性 | 0.20 | 跨分类能力，可用分类名/ID 定位加分 |
| S4 稳定性 | 0.20 | ID 定位 > 内容触发 > 全文匹配 > 索引 |
| S5 意图置信度 | 0.15 | 修改数量、模式特征（符号替换/着色等） |

> **注**：具体评分公式在实现阶段细化。

### 5.4 分层级范围选择

## 七、错误处理 & 边界情况

### 文件层面

| 场景 | 处理 |
|------|------|
| lang 文件夹未配置 | 窗口打开时检测，显示引导提示 |
| lang 文件夹不存在 | 文件列表空状态 + 提示文字 |
| JSON 文件解析失败 | 简单模式显示 "⚠️ 文件格式异常"，高级模式仍可查看原始文本 |
| fancy/ 文件夹不存在 | 首次保存时自动创建 |
| 规则文件名非法字符 | 自动过滤后提示实际文件名 |
| 同名规则文件 | 弹出确认："覆盖 / 重命名" |

### 规则编辑层面

| 场景 | 处理 |
|------|------|
| JSON 语法错误 | 编辑器边框变红，保存禁用，底部显示错误位置 |
| 规则结构不完整 | 验证按钮逐条检查：缺少 aimFile/action、from/to 不成对 |
| 正则非法 | 输入框红色边框 + 实时提示 |
| 空操作列表 | "生成规则"按钮禁用 + hover 提示 |

### 应用层面

| 场景 | 处理 |
|------|------|
| id 不存在（无 fallback） | 规则静默跳过，状态栏提示 |
| 索引 fallback 越界 | 规则静默跳过 |
| 文件写入权限不足 | 弹出错误，列出失败文件列表 |
| 应用中途取消 | 不滚回已修改文件（与现有行为一致） |

### 简单模式特有

| 场景 | 处理 |
|------|------|
| 用户未选文件直接填规则 | 占位提示 + 可用下拉预设 |
| ID 输入非数字 | 实时校验，红色边框，禁用生成 |
| 空操作列表 | 禁用生成，hover 提示 |

## 九、关键设计决策

1. **`conditions` 而非嵌套 trigger**：允许多个 condition，每个有独立 trigger + aim，所有满足才命中。向后兼容旧格式。

2. **ID 定位用 trigger 实现，不引入新语法**：`trigger: { aim: "dataList\.\d+\.id", re: "^10001$" }` + `aim: "[back].name"` 完全等价于假想的 `{id=10001}` 语法，且无需修改 fancy 引擎。

3. **简单模式是高级模式的表单化表达**：用户在简单模式的操作最终生成与高级模式相同格式的 JSON 规则，两者产出完全互通。

4. **智能生成的分层级范围选择**：不自动替用户做决策，而是给每个层级提供选项 + 说明，让用户理解并选择。

5. **规则集文件化**：从 ConfigManager 的巨型 JSON 中解耦，一个文件一个规则集，便于分享、备份和版本管理。
